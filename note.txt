 Oui. La façon propre côté React Router, c’est de créer un petit composant “wrapper” du genre
  ProtectedRoute.

  Son rôle :

  si AuthContext charge encore -> ne rien afficher ou afficher loading
  si user connecté -> afficher la page demandée
  si user non connecté -> rediriger vers /login

  Exemple de fichier :

  frontend/src/auth/ProtectedRoute.tsx

  Contenu logique :

  import { Navigate } from 'react-router-dom';
  import { useAuth } from './AuthContext';

  type ProtectedRouteProps = {
      children: React.ReactNode;
  };

  export function ProtectedRoute({ children }: ProtectedRouteProps) {
      if (isLoading) {
          return null;
      }

      if (!currentUser) {
          return <Navigate to="/login" replace />;
      }

      return children;
  }

  Explication rapide :

  - currentUser : user connecté ou null.
  - isLoading : évite de rediriger trop tôt pendant que /auth/me vérifie le token.
  - Navigate : composant React Router qui redirige.
  - replace : évite d’ajouter la page protégée dans l’historique navigateur.

  Ensuite dans App.tsx, tu protèges les routes concernées.

  Avant :

  <Route path="settings" element={<SettingsPage />} />

  Après :

  <Route
    path="settings"
    element={
      <ProtectedRoute>
        <SettingsPage />
      </ProtectedRoute>
    }
  />

  Pour friends, si une page FriendsPage existe plus tard :

  <Route
    path="friends"
    element={
      <ProtectedRoute>
        <FriendsPage />
      </ProtectedRoute>
    }
  />

  Donc le schéma devient :

  /user va sur /settings
  -> ProtectedRoute regarde AuthContext
  -> si currentUser existe : affiche SettingsPage
  -> sinon : redirige vers /login

  Très important : ça protège seulement le frontend visuellement. Il faut quand même protéger les
  vraies routes backend avec JwtAuthGuard.

  Phrase à retenir :

  ProtectedRoute protège la navigation React.
  JwtAuthGuard protège les données backend.
